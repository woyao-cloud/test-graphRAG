# 第 5 章：向量存储与检索

## 5.1 从文本到向量

向量存储是 RAG 系统的核心基础设施。文档被切分成块之后，下一步就是将这些文本块转换为向量（嵌入），并存储到向量数据库中供后续检索使用。嵌入（Embedding）的本质是将高维语义信息压缩到一个固定长度的向量空间中，使得语义相似的文本在向量空间中距离更近。

嵌入模型的选择直接影响检索的质量和效率。当前主流的嵌入模型可以从以下几个维度进行评估：

- **维度大小**：从 384 维（bge-small）到 3072 维（text-embedding-3-large）。维度越高，表达能力越强，但存储和计算成本也越高。
- **语言支持**：部分模型针对中文优化（如 bge-zh、m3e），通用模型（如 text-embedding-3）则在多语言场景下表现均衡。
- **最大输入长度**：从 512 tokens 到 8192 tokens。块大小需要与模型的最大输入长度匹配。
- **价格与延迟**：开源模型可以在本地部署，成本可控；商业 API 模型质量更高但会产生调用费用。

### 5.1.1 嵌入模型选型建议

| 场景 | 推荐模型 | 维度 | 备注 |
|---|---|---|---|
| 中文通用 | bge-large-zh-v1.5 | 1024 | 中文语义理解优秀，支持本地部署 |
| 多语言通用 | text-embedding-3-large | 3072 | OpenAI 出品，质量最高 |
| 轻量快速 | bge-small-zh-v1.5 | 512 | 适合延迟敏感场景 |
| 开源免费 | gte-large-zh | 1024 | 阿里出品，中英双语 |

## 5.2 向量数据库选型

向量数据库是存储和检索嵌入向量的专门化系统。与传统数据库不同，向量数据库的核心操作是近似最近邻搜索（ANN Search），它在牺牲少量精确度的情况下大幅提升搜索速度。

### 5.2.1 主流向量数据库对比

**FAISS**（Facebook AI Similarity Search）是最成熟的开源向量搜索库。它不是一个完整的数据库系统，而是一个高效的向量索引库，适合在单机场景下使用。FAISS 提供了多种索引类型：

- `IndexFlatIP`：暴力搜索，最准确但最慢，适合小数据集
- `IndexIVFFlat`：倒排索引 + 暴力搜索，速度与精度的权衡
- `IndexHNSWFlat`：分层可导航小世界图，速度快且精度高
- `IndexPQ`：乘积量化，大幅压缩内存占用但精度下降

**Milvus** 是一个分布式向量数据库系统，支持云原生部署。它提供了完整的 CRUD 操作、多租户、混合搜索等功能，适合大规模生产环境。

**LanceDB** 是一个建立在 Lance 列式格式之上的嵌入式向量数据库。它不需要独立的服务器进程，可以直接嵌入到 Python 应用中，适合中小规模场景和原型开发。

**Qdrant** 是一个用 Rust 编写的向量数据库，提供了丰富的过滤功能和高效的性能。

### 5.2.2 混合存储架构

在实际项目中，纯向量检索往往不够——我们需要结合元数据过滤、关键词检索等多种能力。因此，一个成熟的 RAG 系统通常会采用混合存储架构：

**向量数据库 + 元数据索引**：在存储向量的同时，为每个向量关联丰富的元数据（文档来源、更新时间、权限等级等）。检索时先通过向量找到候选集，再通过元数据过滤缩小范围。

**双库架构**：向量数据库负责语义检索，传统搜索引擎（Elasticsearch）负责关键词检索，两者的结果通过 RRF（Reciprocal Rank Fusion）进行融合。

**分层存储**：热数据存储在内存或 SSD 上保证低延迟，冷数据存储在大容量存储上降低成本。根据数据的访问频率自动在层级之间迁移。

## 5.3 索引构建策略

索引构建不仅仅是"把向量存进去"这么简单。一个高效的索引需要考虑以下因素：

### 5.3.1 批量索引 vs 流式索引

**批量索引**在系统启动或数据迁移时使用，一次性处理大量文档，可以充分利用批处理优化的优势（如 GPU 批量推理）。缺点是索引构建期间系统不可用或只能提供部分功能。

**流式索引**在数据持续产生时使用，每次处理一个或一小批文档，索引实时更新。适合数据频繁变动的场景，但整体吞吐量低于批量索引。

生产环境中通常两者结合：初次使用批量索引建立基础索引库，后续通过流式索引保持增量更新。

### 5.3.2 索引参数调优

HNSW 索引有两个关键参数：

- `M`（每个节点的最大连接数）：默认 16。M 越大，检索精度越高但内存占用越大。对于高精度场景可以设为 32-64，对于低内存场景可以设为 8-12。
- `ef_construction`（构建时的搜索范围）：默认 200。值越大构建越慢但质量越高。
- `ef_search`（检索时的搜索范围）：在运行时动态调整。值越大召回率越高但延迟也越高。

### 5.3.3 多向量索引

对于每个文档块，可以使用多个嵌入模型生成多组向量，或者使用同一模型的不同切片策略。多向量索引在检索时可以提供互补的语义覆盖，但也带来了存储成本和检索延迟的增加。

## 5.4 检索优化技术

### 5.4.1 查询扩展

查询扩展（Query Expansion）是在将用户查询送入向量搜索之前，对查询进行增强的技术：

- **同义词扩展**：将查询中的关键术语替换为其同义词，生成多个查询变体
- **翻译扩展**：在多语言场景下，将查询翻译为其他语言后再检索
- **LLM 扩展**：使用语言模型生成查询的补充描述或分解

### 5.4.2 混合搜索

纯向量搜索在处理精确关键词匹配时表现不佳。例如，搜索"产品型号 XYZ-2000"时，向量搜索可能无法精确匹配包含"XYZ-2000"的文档。混合搜索通过将向量搜索与关键词搜索（BM25）结合来解决这个问题：

```
向量搜索得分 × α + 关键词搜索得分 × (1-α) = 最终得分
```

其中 α（融合权重）可以通过网格搜索或贝叶斯优化来确定最优值。

### 5.4.3 分片与分区

当向量数据量超过单机容量时，需要引入分片（Sharding）机制。常用的分片策略包括：

- **哈希分片**：对向量 ID 进行哈希后分配到不同的分片，均匀分布但跨分片查询开销大
- **范围分片**：按时间戳或其他有序维度分片，便于按时间范围过滤
- **产品量化**：将高维向量压缩为短编码，减少内存占用

## 5.5 本章小结

向量存储与检索是 RAG 系统的数据层核心。本章从嵌入模型选型出发，详细介绍了向量数据库的选型考量、索引构建策略和检索优化技术。在实践中，向量存储层的设计应该与上游的文档切分策略和下游的检索策略协同考虑。没有最好的向量存储方案，只有最适合特定业务场景的方案。
