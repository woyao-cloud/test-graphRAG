# 第 12 章：Agentic RAG

## 12.1 从被动检索到主动推理

传统 RAG 系统的检索过程是"一次性的"——用户输入查询，系统检索一次，生成答案。这种模式对于简单的事实查询已经足够，但在面对复杂问题时存在明显局限。例如，当用户问"紫杉醇的供应链是怎样的？"，系统可能需要先查找紫杉醇的生产商，再找原料药供应商，再找分销商——这需要多步推理，而不是一步检索。

Agentic RAG（智能体增强检索）将大语言模型的智能体能力与 RAG 系统结合起来，使系统具备"思考→行动→观察→再思考"的循环推理能力。智能体可以根据当前信息自主决定下一步动作（检索、查询知识图谱、计算、总结），直到收集到足够的信息来回答问题。

## 12.2 ReAct 模式

ReAct（Reasoning + Acting）是 Agentic RAG 的基础模式，它交替执行推理（Reasoning）和行动（Acting）两个步骤：

**推理**：智能体分析当前已知信息和用户问题，决定下一步需要什么信息。
**行动**：基于推理结果，执行具体的操作——可能是向量检索、知识图谱查询或调用外部工具。
**观察**：获取行动的结果，将其添加到上下文记忆中。
**循环**：重复推理-行动-观察循环，直到收集到足够的信息或达到最大步数限制。
**最终答案**：基于收集到的所有信息生成最终回答。

ReAct 模式的核心价值在于透明性——用户可以看到系统一步步的推理过程，这增加了答案的可信度和可解释性。

## 12.3 Corrective RAG（CRAG）

CRAG（Corrective RAG）是 Agentic RAG 的一种重要变体，它专注于提升检索质量。与一次性检索不同，CRAG 在每次检索后都会评估结果质量，并根据评估结果决定下一步动作：

- **正确（Correct）**：检索结果与问题高度相关，直接生成答案
- **模糊（Ambiguous）**：检索结果部分相关，补充检索后生成答案
- **错误（Incorrect）**：检索结果不相关，重写查询后重新检索，最多重试 N 次

CRAG 的评估机制是核心挑战。常用的评估方法包括：
- **相关性阈值**：将检索分数的平均值与预设阈值比较
- **实体覆盖率**：检查检索结果中是否包含问题中的关键实体
- **LLM 评估**：使用语言模型判断检索结果是否与问题相关

当检索多次失败后，CRAG 还包含一个兜底策略——"不知道"回答机制，避免在信息不足时生成误导性答案。

## 12.4 Self-RAG

Self-RAG 是一种更加自省的 Agent 模式。它不仅评估检索结果的整体相关性，还逐片段评估每个检索结果的有用性，并在此基础上过滤和调整生成过程。

Self-RAG 引入了一个"反思标记"（Reflection Token）的概念。对于每个检索到的文本片段，模型会生成两个标记：
- **IsRel（相关性标记）**：该片段是否与问题相关
- **IsSup（支持标记）**：该片段是否支持或反驳当前正在生成的答案

基于这些标记，Self-RAG 可以有选择地包含或排除检索片段，甚至在生成过程中动态调整。这种方式能够有效减少检索噪声的影响，提升生成答案的忠实度（Faithfulness）。

## 12.5 多 Agent 协作

当任务复杂到一定程度，单个 Agent 的能力可能不够用。多 Agent 协作架构将复杂的检索任务分解为多个子任务，由不同的 Agent 并行或串行完成。

一个典型的多 Agent RAG 架构包含以下角色：

**编排器（Orchestrator）**：接收用户问题，将其分解为子任务，分配给合适的 Worker Agent，然后汇总结果。编排器需要判断问题的复杂度——简单问题直接调用检索，复杂问题才启动多 Agent 协作。

**知识图谱 Worker**：专门负责知识图谱查询，擅长处理实体关系类问题。

**向量检索 Worker**：专门负责向量语义检索，擅长处理文本匹配类问题。

**网页搜索 Worker**：负责联网获取最新信息，适合处理时效性要求高的问题。

**推理 Worker**：负责多步推理和计算，如数学计算、逻辑推理等。

多 Agent 协作的优势是模块化和专业化——每个 Agent 专注于自己擅长的领域，整体系统可以通过增加新的 Worker 来扩展能力边界。

## 12.6 Agent 模式选择指南

| 问题类型 | 推荐模式 | 说明 |
|---|---|---|
| 简单事实查询 | 传统 RAG | 一次性检索即可，无需 Agent 开销 |
| 多步推理 | ReAct | 逐步推理，串行执行多步动作 |
| 检索质量不稳定 | CRAG | 评估检索质量，自动重试和纠错 |
| 高精度要求 | Self-RAG | 逐片段评估，精确控制生成 |
| 复杂跨领域任务 | 多 Agent | 分解为子任务，并行执行 |

需要注意的是，Agent 模式虽然强大，但带来了额外的延迟和成本。在实际部署中，建议设置一个"复杂度分类器"，仅在需要时才启用 Agent 模式，简单问题仍然走传统 RAG 流程。

## 12.7 本章小结

Agentic RAG 将大语言模型的推理能力与 RAG 系统的检索能力深度结合，使系统具备了多步推理、自我纠错和任务分解的能力。本章介绍了 ReAct、CRAG、Self-RAG 和多 Agent 协作四种 Agent 模式，每种模式适用于不同类型的问题复杂度。选择合适的 Agent 模式是在响应质量和系统开销之间做权衡。

完整的可运行代码示例见 `demos/ch12/agentic_rag.py`，它演示了 ReAct、CRAG 和多 Agent 协作三种模式的完整实现。
