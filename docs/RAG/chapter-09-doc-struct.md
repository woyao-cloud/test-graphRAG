# 第 4 章：多源文档处理管线

## 4.1 文档处理在 RAG 中的角色

RAG 系统的质量上限在索引阶段就已经决定了。无论检索策略多先进、生成模型多强大，如果输入文档没有被正确处理，后续所有环节的效果都会受到根本性的制约。文档处理管线（Document Processing Pipeline）是 RAG 系统与原始数据之间的第一道关卡，它负责将各种格式、各种来源的原始文档转化为结构化的、可检索的文本块。

企业环境中的文档生态远比想象的复杂。一个典型的中型企业在日常运营中会产生数十种格式的文档：PDF 格式的产品手册、Word 格式的合同文档、Markdown 格式的技术文档、HTML 格式的网页帮助中心、CSV/JSON 格式的数据库导出、甚至扫描件图片。每种格式都有其独特的解析挑战。文档处理管线的核心目标就是建立一个统一的处理框架，将这些异构的输入转化为标准化的输出。

本章将深入讲解文档处理管线的每一个环节：从多格式解析、文本清洗、编码检测，到文档切分和元数据管理，并给出完整的代码实现。

## 4.2 多格式文档解析

### 4.2.1 统一文档模型

要处理多种格式，首先需要定义一个统一的数据模型，将所有格式的解析结果映射到同一个结构上：

```python
@dataclass
class Document:
    id: str                # 唯一标识
    title: str             # 文档标题
    content: str           # 正文内容
    source_path: str       # 源文件路径
    source_format: str     # 原始格式（pdf/md/html/txt）
    metadata: dict         # 格式特定的元数据
    created_at: str        # 处理时间戳
```

这个模型的核心思想是**格式无关化**——无论输入是 PDF 还是 Markdown，最终都输出统一的 Document 对象。差异被封装在 metadata 字段中，也可以根据需要在后续环节中忽略。

### 4.2.2 各格式解析要点

**PDF 解析**是文档处理中最复杂的场景。PDF 本质上是一种页面描述语言，而不是结构化文本。解析方式的选择取决于 PDF 的类型：

- **文本型 PDF**（由 Word/LaTeX 等工具生成）：使用 PyMuPDF（fitz）或 pdfplumber 可以直接提取文本和表格。这类 PDF 内部包含文本对象的定位信息，提取质量较高。
- **扫描型 PDF**（由扫描仪生成的无文本层 PDF）：需要 OCR 引擎支持。PaddleOCR 在中文场景下表现优秀，Tesseract 则是一个成熟的开源选择。
- **复杂布局 PDF**（包含多栏、表格、页眉页脚）：Unstructured 库提供了策略化的解析方式，可以自动识别文档元素类型（标题、段落、表格、列表）。

**Markdown 解析**相对简单但需要保留结构信息。标题层级（H1-H6）对于后续的文档感知切分非常关键——它天然定义了文档的章节边界。代码块需要特殊处理，通常保留原样但在切分时尽量不拆分代码块。

**HTML 解析**的核心是从结构化标记中提取纯文本内容。BeautifulSoup 是最常用的工具，但需要注意几点：去除 script 和 style 标签中的非内容文本；识别并保留标题（h1-h6）的层级信息；处理列表和表格的结构化内容。Trafilatura 是一个专注于正文提取的库，对新闻文章和博客类页面效果很好。

**JSON/CSV 数据**则需要将结构化字段转换为自然语言描述。例如，一条产品记录可以转换为"产品名称为 xxx，供应商为 xxx，价格为 xxx 元"这样的文本。

### 4.2.3 编码检测与处理

中文文档经常遇到编码问题——一个来自 Windows 系统的 TXT 文件可能是 GBK 编码，而 Linux 系统生成的是 UTF-8。直接以错误编码读取会导致乱码。解决方案是使用 chardet 或 charset-normalizer 进行编码检测：

```python
import chardet

def detect_and_read(file_path):
    raw = open(file_path, "rb").read()
    result = chardet.detect(raw)
    encoding = result["encoding"]
    confidence = result["confidence"]
    if confidence < 0.7:
        encoding = "utf-8"  # 低置信度时回退
    return raw.decode(encoding, errors="replace")
```

生产环境中建议将检测到的编码记录到元数据中，便于后续排查问题。

## 4.3 数据清洗管线

原始文本中充斥着各种噪声，数据清洗的目标是在保留语义的前提下最大限度地去除无关内容。一个典型的数据清洗管线由一系列可配置的规则组成：

1. **零字节去除**：某些系统生成的文本中会混入 `\x00` 字符，会导致后续处理异常。
2. **Unicode 规范化**：将全角/半角、组合字符等统一为 NFC 或 NFKC 形式。中文场景下，全角英文字母和数字需要转换为半角。
3. **空白字符压缩**：将多个连续的空格、制表符、换行符压缩为单个空格。
4. **空行去除**：删除不包含任何可见字符的行。
5. **特殊字符处理**：控制字符、不可见字符的去除或替换。

这些规则应该设计为可插拔的，便于根据不同文档类型启用或禁用：

```python
class DataCleaner:
    def __init__(self):
        self.rules = []
    
    def add_rule(self, name, rule_fn):
        self.rules.append((name, rule_fn))
    
    def clean(self, doc):
        for name, rule in self.rules:
            doc.content = rule(doc.content)
        return doc
```

## 4.4 文档切分策略

文档切分（Chunking）是文档处理管线中最关键也最容易被忽视的环节。切分策略直接决定了检索的召回率和准确率。

### 4.4.1 固定大小切分

最简单的策略是按固定字符数切分，并引入重叠（overlap）来避免上下文断裂：

- **优点**：实现简单、块大小可控、性能可预测
- **缺点**：可能在语义完整的中断处切分，导致信息不完整
- **适用场景**：对切分质量要求不高的快速原型

重叠窗口的设置需要权衡：过小的重叠（<10%）无法有效连接上下文，过大的重叠（>30%）会导致大量冗余。推荐的默认值是 chunk_size=500，overlap=50（即 10% 的重叠率）。

### 4.4.2 递归切分

递归切分（Recursive Chunking）是一种更智能的策略，它按层级尝试不同的分隔符：

1. 首先尝试按段落（\n\n）切分
2. 如果段落太长，按句子（。/!/?）切分
3. 如果句子仍然太长，按标点或固定长度切分

这种策略的核心优势是尽可能保持语义完整性。一个完整的段落比一个被随意切分的文本片段更容易被检索系统和语言模型理解。

### 4.4.3 语义切分

语义切分（Semantic Chunking）利用嵌入向量的相似度来检测主题边界。当相邻句子的嵌入向量相似度低于某个阈值时，认为发生了主题转换，在此处切分。这种方法的效果最好，但计算成本也最高。

实践中常见的做法是混合使用：先用递归切分得到候选块，再用语义相似度对相邻块进行合并或拆分。

### 4.4.4 文档感知切分

如果能利用文档的结构信息（如 Markdown 标题、PDF 的书签），切分质量可以进一步提升。**文档感知切分**（Document-Aware Chunking）的基本思路是：

1. 将文档解析为结构树（标题 → 章节 → 段落）
2. 以自然章节为单位进行切分
3. 将章节标题作为上下文元数据附加到每个块上

```python
# 示例：基于 Markdown 标题的文档感知切分
def document_aware_chunk(markdown_text, max_chars=500):
    lines = markdown_text.split("\n")
    chunks = []
    current_section = []
    current_header = ""
    
    for line in lines:
        if line.startswith("#"):
            # 保存当前章节
            if current_section:
                chunks.append({
                    "header": current_header,
                    "content": "\n".join(current_section)
                })
            current_header = line
            current_section = [line]
        else:
            current_section.append(line)
    
    return chunks
```

这种方式检索到的每个块都携带了章节标题信息，在生成答案时可以作为上下文来源的引用。

## 4.5 元数据管理

文档处理管线不仅要提取文本内容，还需要构建丰富的元数据体系。好的元数据设计能够显著提升检索的精准度：

- **来源元数据**：文件路径、文档类型、创建时间、最后修改时间
- **结构元数据**：章节层级、段落位置、是否包含代码块/表格/图片
- **处理元数据**：使用的清洗规则列表、切分策略、编码信息
- **业务元数据**：文档所属部门、文档分类、权限等级、版本号

在切分块中继承文档的元数据信息，可以支持基于元数据的过滤检索。例如，只检索某个部门的文档，或者只检索最新版本的文档。

## 4.6 完整管线设计

一个生产级的文档处理管线通常采用流水线（Pipeline）架构，每个阶段专注于一个职责：

```
输入文件 → 格式检测 → 多格式解析 → 数据清洗 → 编码标准化 → 文档切分 → 元数据标记 → 索引存储
```

每个阶段都应该设计为独立的可组合单元。这样做的好处是：
- **可测试性**：每个阶段的输出可以单独验证
- **可扩展性**：可以在任意阶段插入新的处理器（如自定义的清洗规则）
- **可观测性**：每个阶段都可以记录处理时间和异常

管线的容错设计同样重要。对于批量处理场景，单个文件的处理失败不应该影响整个批次的处理。建议为每个文件使用独立的 try-catch 块，并将错误信息记录到处理日志中。

## 4.7 本章小结

文档处理管线是 RAG 系统的地基。地基不牢，上层建筑的质量就无从保证。本章详细讲解了从多格式解析到文档切分的完整流程，每个环节都有其专门的技术挑战和最佳实践。在实际项目中，建议为文档处理管线预留充足的开发和调试时间，因为它直接影响后续所有环节的效果上限。
