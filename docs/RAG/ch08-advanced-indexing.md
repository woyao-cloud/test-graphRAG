# 第 8 章：高级索引策略

## 8.1 索引优化的必要性

随着知识库规模的增长，RAG 系统的检索延迟和资源消耗会迅速上升。一个拥有 100 万文档的系统，如果缺乏有效的索引优化策略，检索延迟可能从毫秒级退化到秒级，内存占用可能达到几十 GB。高级索引策略就是在这种背景下产生的——在不显著牺牲召回率的前提下，通过精巧的数据结构和算法设计来压缩索引体积、加速检索过程。

## 8.2 向量量化技术

向量量化（Vector Quantization）是压缩向量存储的核心技术。它将高维浮点向量映射为低精度或低维度的表示，从而大幅减少内存占用。

### 8.2.1 标量量化（SQ）

标量量化将 32 位浮点数映射为 8 位整数。对于每个维度，计算所有向量在该维度上的取值范围，然后将连续的值映射到 256 个离散的整数桶中。这样做的效果是将每个向量从 4×D 字节压缩为 1×D 字节，压缩比达到 4:1，而召回率的下降通常在 1-3% 以内。

### 8.2.2 乘积量化（PQ）

乘积量化将高维向量分割为若干子向量，对每个子空间独立进行聚类和量化。例如，将 1024 维向量分割为 8 个 128 维的子空间，每个子空间用 256 个聚类中心（8 位编码）表示。这样每个向量只需要 8×8=64 位（8 字节）的存储，而原来是 1024×4=4096 字节，压缩比高达 512:1。

PQ 的代价是检索精度会有所下降，且索引构建时间变长。但在大规模场景下（千万级以上），PQ 几乎是唯一可行的内存方案。

### 8.2.3 量化策略选择

| 策略 | 压缩比 | 精度损失 | 构建速度 | 适用规模 |
|---|---|---|---|---|
| 无量化 (float32) | 1:1 | 无 | 快 | <10 万 |
| 标量量化 (SQ8) | 4:1 | ~1% | 较快 | 10-100 万 |
| 乘积量化 (PQ) | 可达 512:1 | 1-5% | 慢 | 100 万以上 |
| OPQ（优化乘积量化） | 可达 512:1 | <PQ | 较慢 | 100 万以上 |

## 8.3 索引类型与选择

### 8.3.1 平面索引（Flat）

平面索引是最简单的索引方式——暴力计算查询向量与所有存储向量的相似度。它的精度是最高的（100% 召回），但延迟随规模线性增长。适用于小数据集（<1 万条）或对精度有极致要求的场景。

### 8.3.2 IVF（倒排文件索引）

IVF 通过聚类将向量空间分割为多个 Voronoi 单元格。检索时只在查询向量最近的 N 个单元格中搜索，而不是搜索全部向量。关键参数 `nprobe` 控制搜索的单元格数量——nprobe 越大，召回率越高但延迟也越高。

IVF 的参数调优是在召回率和延迟之间做权衡。推荐的调优路径是：先固定 nlist（聚类中心数）为 sqrt(N)，然后从 nprobe=1 开始逐步增加，观察召回率曲线的饱和点。

### 8.3.3 HNSW（分层可导航小世界图）

HNSW 是当前最流行的 ANN 索引算法，它在精度和速度之间取得了极佳的平衡。基本原理是构建一个多层图结构：顶层稀疏连接，底层密集连接。搜索时从顶层开始逐层下降，每层使用贪婪搜索。

HNSW 的三个关键参数：
- `M`：每个节点在底层的最大连接数。M 越大图越稠密，精度越高但内存占用和构建时间也越大。推荐值 16-32。
- `ef_construction`：构建时的动态列表大小。控制索引质量，推荐值 100-500。
- `ef_search`：检索时的动态列表大小。控制检索精度，可以在运行时动态调整。

### 8.3.4 索引选择决策树

```
数据量 < 1 万 → 平面索引（Flat，精度优先）
数据量 < 10 万 → HNSW（精度和速度平衡）
数据量 < 100 万 → IVF+HNSW 混合
数据量 > 100 万 → IVF+PQ 或 HNSW+PQ（内存优先）
延迟敏感场景 → HNSW（ef_search 调低）
精度敏感场景 → Flat 或 HNSW（ef_search 调高）
```

## 8.4 过滤预检与分段索引

### 8.4.1 基于元数据的过滤

在实际业务中，检索往往伴随着过滤条件——"只搜索 2024 年的文档"、"只搜索属于客户支持类别的文档"。如果向量索引感知不到这些过滤条件，就需要在检索后重新过滤，可能浪费大量计算。

实现过滤感知的索引有几种方式：
- **ID 过滤**：为每个向量分配可过滤的 ID，检索时跳过被过滤的 ID
- **分段索引**：为每种元数据组合创建独立的索引段
- **BitMap 过滤**：使用位图标记每个文档的元数据属性，在检索时快速过滤

分段索引在元数据过滤频繁变动时效果更好，但段的管理和合并需要额外开销。ID 过滤实现简单但过滤效率随被过滤比例线性下降。

### 8.4.2 时间感知索引

对于有明确时间维度的知识库，时间衰减（Time Decay）是一种有效的前置过滤策略。离当前时间越近的文档获得越高的基础权重。实现方式包括：

- **时间分段**：按年/月/周分段索引，搜索时优先搜索最近的时间段
- **时间加权**：在向量相似度分数上叠加时间衰减因子
- **时间过滤**：在查询时强制指定时间范围

## 8.5 增量索引与索引合并

### 8.5.1 分段合并策略

增量索引自然会产生索引碎片——小批量的新数据被添加到独立的索引段中，导致检索时需要搜索多个段。定期合并段可以优化搜索性能：

- **按大小分层的段管理**：小的段频繁合并，大的段少合并
- **段合并时机**：段数量超过阈值（如 10 个）或段大小差异过大时触发合并
- **合并策略**：使用 Size-tiered 策略（类似 Cassandra 或 Elasticsearch）

### 8.5.2 软删除与硬删除

当文档需要从知识库中移除时，软删除（标记删除而不是物理删除）可以避免重建索引的高昂成本。软删除的文档在检索时被跳过，但占用的存储空间不会被立即释放。定期执行硬删除和索引重建来回收空间。

## 8.6 本章小结

高级索引策略是 RAG 系统从原型走向生产的关键技术。本章介绍了向量量化（SQ、PQ）、索引类型选择（Flat、IVF、HNSW）、过滤预检和增量索引管理等技术。在实践中，索引策略的选择需要结合数据规模、延迟要求、精度要求和硬件资源进行综合权衡。没有最优的索引方案，只有最适合特定场景的方案。
