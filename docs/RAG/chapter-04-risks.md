# 第 14 章：性能与内存管理

## 14.1 RAG 系统的性能瓶颈

RAG 系统的性能瓶颈通常出现在四个环节：

1. **Embedding 计算**：将文本转换为向量需要调用嵌入模型，这是在 CPU 上较为耗时的操作。批量 Embedding 虽然在 GPU 上很快，但在 CPU 环境下单次推理可能需要 50-500ms。
2. **向量检索**：在大规模向量库中执行 ANN 搜索，HNSW 图的搜索延迟通常在 10-50ms，但如果索引参数配置不当可能上升到数百毫秒。
3. **Reranker 重排**：Cross-Encoder 比 Bi-Encoder 慢 1-2 个数量级，50 个候选的重排可能消耗 200-1000ms。
4. **LLM 生成**：生成阶段是整条链路中最慢的环节，小模型通常需要 1-3 秒，大模型可能需要 5-10 秒以上。

性能优化的核心思路是"分层缓存、并行执行、按需计算"——尽量减少重复计算，尽量并行执行独立任务，尽量延迟非必要的计算。

## 14.2 缓存策略

### 14.2.1 Embedding 缓存

Embedding 计算是 RAG 系统中重复率最高的操作。同一个文本块可能在不同的检索请求中多次被查询，如果每次都重新计算 Embedding，将是巨大的浪费。

Embedding 缓存的基本实现：
```python
class EmbeddingCache:
    def __init__(self, max_size=10000):
        self.cache = {}
        self.max_size = max_size
    
    def get_or_compute(self, text, compute_fn):
        key = md5(text).hexdigest()
        if key in self.cache:
            return self.cache[key]
        embedding = compute_fn(text)
        if len(self.cache) >= self.max_size:
            # LRU 淘汰
            self._evict()
        self.cache[key] = embedding
        return embedding
```

缓存的淘汰策略通常选择 LRU（最近最少使用）或 LFU（最不经常使用），根据访问模式选择。LRU 适用于访问模式有局部性的场景，LFU 适用于长尾频繁访问的场景。

### 14.2.2 多级缓存

在生产环境中，单一级别的缓存往往不够。推荐使用多级缓存架构：

- **L1 内存缓存**：速度最快（微秒级），容量最小（GB 级）。使用 LRU 策略，TTL 较短。
- **L2 Redis/磁盘缓存**：速度中等（毫秒级），容量较大（TB 级）。持久化存储，TTL 较长。
- **L3 远端缓存**（可选）：在分布式部署中共享缓存。

多级缓存的访问顺序是 L1 → L2 → 计算。命中 L1 最快，L2 次之，两层都未命中时才执行计算并回填。

### 14.2.3 查询结果缓存

对于高频重复查询（如 FAQ），可以缓存完整的查询结果。缓存键需要考虑用户问题的语义等价性——"恒瑞医药生产哪些药品"和"恒瑞医药有哪些药品"应该命中同一个缓存。

语义缓存的一种实现方案是：使用 Embedding 相似度来判断新问题是否与缓存中的某个问题语义等价。如果相似度超过阈值（如 0.95），直接返回缓存的答案。

## 14.3 并发执行

### 14.3.1 多路并行检索

多路召回中的不同检索策略（稠密检索、稀疏检索、KG 检索）之间没有依赖关系，可以并行执行。

```python

```

在 Python 中，对于 I/O 密集型任务（如网络请求、数据库查询），使用 asyncio 或线程池可以实现接近线性的加速。对于 CPU 密集型任务（如 Embedding 计算），多进程是更好的选择。

### 14.3.2 请求级别并行

对于高并发场景，请求级别的并行处理框架（如 FastAPI + asyncio）可以显著提升吞吐量。关键要点是确保所有共享资源（缓存、连接池）都是线程安全的。

## 14.4 连接池管理

数据库连接（向量数据库、关系数据库、图数据库）的建立和销毁是昂贵的操作。连接池通过维护一组预先建立的连接来减少连接开销。

连接池的核心参数：
- **最小连接数**：预先建立的连接数量，保证基础吞吐量
- **最大连接数**：限制系统资源占用，防止连接泄露
- **超时时间**：等待可用连接的最长时间
- **健康检查**：定期检查连接的有效性，自动修复断裂连接

连接池的关键指标监控：活跃连接数、等待队列长度、连接获取延迟。如果等待队列持续增长，说明连接池容量不足，需要扩容。

## 14.5 Token 预算管理

LLM API 调用是按 token 计费的。在没有预算管理的情况下，一个失控的 RAG 系统可能在几小时内消耗完月度预算。

Token 预算管理的核心策略：
- **日预算限制**：设定每日最大 token 消耗，超出后降级到纯检索（不使用 LLM 生成）或返回错误
- **请求级预算**：单次请求的 token 消耗上限，防止过长上下文导致单个请求成本失控
- **用户级预算**：在多租户场景下，为每个用户/应用分配独立的预算
- **预算预警**：当消耗达到预算的 80%、90%、100% 时触发不同级别的告警

## 14.6 性能监控

性能监控是持续优化的前提。一个标准的 RAG 性能监控体系应该覆盖以下指标：

**延迟指标**：
- P50/P95/P99 分位数延迟
- 各子阶段（Embedding、检索、重排、生成）的耗时分布
- 缓存命中率

**吞吐量指标**：
- QPS（每秒查询数）
- 并发请求数
- 资源利用率（CPU、内存、网络）

**成本指标**：
- 单次查询平均 token 消耗
- 每日 API 调用次数和成本
- 缓存节省的成本

**质量指标**：
- 检索召回率（离线评估）
- 答案准确率（人工采样评估）
- 用户反馈（点赞/点踩）

## 14.7 本章小结

性能与内存管理是 RAG 系统从原型走向生产的关键工程环节。本章介绍了 Embedding 缓存、多级缓存、并发执行、连接池管理、Token 预算管理和性能监控等核心技术。性能优化的本质是在延迟、成本和质量之间寻找最佳平衡点——没有绝对的最优，只有针对特定业务场景的最优解。
