# 第 9 章：知识图谱检索

## 9.1 知识图谱在 RAG 中的定位

传统的向量检索擅长在非结构化文本中寻找语义相似的片段，但当问题涉及多步推理、精确关系查询时——例如"恒瑞医药通过哪些分销商将药品送到北京协和医院？"——向量检索就力不从心了。这类问题需要理解实体之间的多跳关系，而这正是知识图谱的强项。

知识图谱在 RAG 系统中扮演着"结构化记忆"的角色。它不是替代向量检索，而是与向量检索形成互补：向量检索负责宽泛的语义匹配，知识图谱负责精确的关系推理。两者结合的混合检索架构，能够覆盖从开放域问答到结构化查询的完整问题类型。

## 9.2 实体链接

### 9.2.1 命名实体识别（NER）

知识图谱检索的第一步是从用户查询中识别出命名实体。例如，从"恒瑞医药的紫杉醇供应商是谁？"中识别出"恒瑞医药"（药企）和"紫杉醇"（药物）。常用的 NER 方法包括：

- **基于词典的 NER**：预先构建实体词典，通过最大正向匹配或 AC 自动机在文本中匹配实体。速度快但无法处理未见实体。
- **基于模型的 NER**：使用预训练语言模型（如 BERT）进行序列标注。泛化能力强但需要标注数据。
- **LLM-based NER**：将实体识别视为生成任务，通过提示词直接让 LLM 提取实体。灵活但成本高。

生产环境中通常采用"词典+模型"的混合策略：先用词典做快速匹配，再用模型处理未被词典覆盖的实体。

### 9.2.2 实体消歧

实体消歧解决的是"同名不同义"的问题。例如，"苹果"可以指水果也可以指科技公司。消歧的常用方法：

- **上下文消歧**：利用实体周围的上下文词汇来判断其所属的类别
- **社区消歧**：通过实体在知识图谱中的邻居关系来判断——如果"苹果"的邻居有"库克"和"iPhone"，则它应该是科技公司
- **Prompt 消歧**：在检索前用 LLM 对歧义实体进行消歧判断

## 9.3 图遍历

### 9.3.1 自我网络（Ego Network）

EGO 网络是以某个实体为中心，向外扩展 N 跳的子图。这是知识图谱检索中最基础也是最有用的操作。给定一个实体，EGO 网络返回该实体直接或间接关联的所有实体和关系。

EGO 网络的深度选择是关键：深度为 1 只能获取直接邻居，适合简单的事实查询（如"恒瑞医药生产哪些药品？"）；深度为 2-3 可以捕获间接关系，适合多跳推理（如"恒瑞医药的药品供应到哪些医院？"）。

### 9.3.2 最短路径查找

最短路径用于回答"实体 A 和实体 B 之间如何关联"这类问题。例如，"华海药业和北京协和医院之间有什么关系？"——路径可能为：华海药业 → 供应紫杉醇 API → 恒瑞医药 → 生产注射用紫杉醇 → 北京协和医院（使用）。

在知识图谱中查找最短路径通常使用 BFS（广度优先搜索），因为它天然保证找到的路径是最短的。对于大规模图谱，需要引入剪枝策略来控制搜索范围：

- **方向约束**：只沿出边或入边搜索
- **关系类型过滤**：只搜索特定类型的关系
- **跳数限制**：设定最大搜索深度（通常不超过 5 跳）

### 9.3.3 子图匹配

对于更复杂的查询模式（如"找出所有同时使用恒瑞医药和齐鲁制药药品的医院"），需要子图匹配（Subgraph Matching）技术。子图匹配将自然语言查询转化为图模式，然后在知识图谱中匹配满足该模式的所有子图。

## 9.4 KG 增强的检索

### 9.4.1 检索流程

KG 增强的检索流程通常分为以下步骤：

1. **实体链接**：从用户查询中识别实体
2. **图遍历**：以识别到的实体为起点，进行 N 跳图遍历
3. **上下文构建**：将遍历得到的实体和关系格式化为结构化上下文
4. **答案生成**：将结构化上下文与原始文本段落一起送入 LLM 生成答案

### 9.4.2 上下文格式化

将图结构转化为 LLM 能理解的自然语言是关键步骤。一个有效的格式化策略是：先按实体分组，每个实体下罗列其关系和目标实体，形成一个层级化的上下文描述。

```python
def format_graph_context(entities, relations):
    context = []
    for entity in entities:
        context.append(f"实体: {entity.name} ({entity.type}) - {entity.description}")
        for rel in entity.relations:
            context.append(f"  -> [{rel.type}] {rel.target_name}")
    return "\n".join(context)
```

### 9.4.3 混合检索策略

在实际部署中，KG 检索通常不是独立使用的，而是与向量检索结合：

- **KG 优先**：如果查询中识别到明确的实体，先执行 KG 检索
- **向量兜底**：如果 KG 检索没有返回结果，回退到向量检索
- **融合模式**：KG 检索结果和向量检索结果通过 RRF 或加权融合合并

## 9.5 本章小结

知识图谱检索为 RAG 系统提供了结构化推理能力，与向量检索形成互补。本章介绍了从实体链接、图遍历到 KG 增强检索的完整流程。知识图谱检索的核心价值在于精确——当问题涉及明确的关系和多跳推理时，KG 检索比纯文本检索更可靠、更可解释。

完整的可运行代码示例见 `demos/ch09/kg_retrieval.py`，它演示了实体链接、EGO 网络遍历、最短路径查找和 KG 增强混合检索的完整实现。
