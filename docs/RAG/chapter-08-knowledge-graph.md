# 第 13 章：知识图谱构建与应用落地

## 13.1 从文档到结构化知识

知识图谱（Knowledge Graph, KG）的本质是以图结构来组织和表示知识。与纯文本的向量表示不同，知识图谱中的每个节点代表一个实体，每条边代表实体之间的关系。这种表示方式天然适合关系推理和多跳查询。

知识图谱构建（Knowledge Graph Construction, KGC）是将非结构化文本转化为结构化图数据的过程。一个完整的 KGC 流水线包括实体抽取、关系抽取、图谱存储和质量控制四个主要阶段。

## 13.2 Schema 设计

在开始构建知识图谱之前，首先要设计 Schema——定义实体类型、关系类型和约束规则。Schema 是知识图谱的骨架，决定了图谱的表达能力上限。

```
实体类型：
  organization（组织/企业）
  pharma_company（制药企业）
  distributor（分销商）
  hospital（医院）
  regulator（监管机构）
  drug（药品）
  chemical（化学物质）
  disease（疾病）
  department（科室）
  place（地点）

关系类型（含域/范围约束）：
  生产：organization → drug（组织生产药品）
  供应API：organization → organization（供应商向采购方提供原料）
  分销：organization → organization（分销商代理药品）
  使用：organization → drug（医疗机构使用药品）
  监管：organization → drug（监管机构审批药品）
  治疗：drug → disease（药品治疗的疾病）
  位于：organization → place（组织所在地）
```

Schema 设计需要遵循几个原则：
- **适度泛化**：类型过多会增加构建难度，过少则会丢失信息。10-20 个实体类型和 10-15 个关系类型是一个合理的起点。
- **域/范围约束**：定义关系的源和目标类型，可以在抽取时进行校验，减少噪声。
- **可扩展性**：Schema 不是一成不变的，需要预留扩展空间。

## 13.3 实体抽取

### 13.3.1 基于词典的抽取

基于词典的实体抽取是最简单直接的方法。预先构建领域实体词典，通过字符串匹配在文本中找出实体出现的位置。

词典抽取的关键问题是如何处理实体别名和嵌套实体。例如，"恒瑞医药"可能被简称为"恒瑞"，而"注射用紫杉醇"的内部包含"紫杉醇"。

解决策略：
- **最大正向匹配**：优先匹配最长的实体名，避免被短名碎片化切割
- **AC 自动机**：多模式匹配算法，可以在 O(n) 时间内完成所有词典条目的一次匹配
- **优先级排序**：长匹配优先于短匹配，明确名优先于别名

词典抽取的优点是速度快、可解释性强，缺点是冷启动成本高（需要人工构建词典）和无法处理未登录词。

### 13.3.2 基于 LLM 的抽取

LLM 可以理解上下文语义，抽取开放类型的实体。通过精心设计的提示词，可以引导 LLM 从文本中提取符合 Schema 约束的实体。

```python
prompt = f"""
从以下文本中提取实体，实体类型包括：organization、drug、chemical、disease。
输出格式为JSON列表，每个实体包含name、type、description字段。

文本：{text}
"""
```

LLM 抽取的优势是不需要预定义词典，能处理开放领域的实体。缺点是成本高、速度慢、可能存在幻觉。

### 13.3.3 混合抽取策略

生产环境中通常采用"词典+LLM"的混合策略：
1. 用词典做快速匹配，覆盖高频核心实体
2. 用 NLP 模型（如 BERT-NER）补充词典未覆盖的实体
3. 用 LLM 处理复杂场景（如嵌套实体、歧义消解）

这种分层策略在成本和质量之间取得了较好的平衡。

## 13.4 关系抽取

### 13.4.1 基于触发词的关系抽取

触发词（Trigger Word）是文本中暗示某种关系存在的词语或模式。例如，"生产"暗示生产关系，"分销"暗示分销关系。

基于触发词的抽取流程：
1. 对每种关系类型，定义一组触发词（如"生产"→ ["生产", "制造", "研发"]）
2. 在文本中定位触发词的出现位置
3. 在触发词前后窗口内查找符合 Schema 约束的实体对
4. 输出实体对和关系类型

触发词方法实现简单、速度快，但覆盖范围受限于触发词列表的质量。

### 13.4.2 Schema 校验

关系抽取结果需要经过 Schema 校验来过滤不合理的关系。例如，"生产"关系的源类型应该是 organization，目标类型应该是 drug。如果抽取到的实体对不符合这个约束，应该被过滤掉。

校验时需要注意的是类型继承关系——pharma_company 是 organization 的子类型，当 Schema 要求源类型为 organization 时，pharma_company 应该通过校验。

## 13.5 图谱质量检查

构建完成的知识图谱需要经过一系列质量检查才能投入使用：

**孤立实体检查**：没有任何关系连接的实体。孤立实体无法参与图遍历，通常说明抽取过程遗漏了关系或该实体本身是噪声。

**Schema 违规检查**：关系两端的实体类型不符合 Schema 定义。违规通常说明关系抽取有误。

**重复实体检查**：同一实体被以不同名称多次添加（如"恒瑞医药"和"恒瑞医药有限公司"）。需要通过实体对齐（Entity Alignment）技术来合并。

## 13.6 Graph + Vector 混合查询

构建知识图谱的最终目的是使用。在落地应用中，纯图谱查询和纯向量检索都有局限性，因此通常采用混合查询模式：

1. **实体识别**：从用户问题中识别出知识图谱中的实体
2. **图谱查询**：以识别到的实体为起点，在图谱中进行 N 跳邻居查询
3. **文本匹配**：同时使用关键词或向量检索来补充图谱覆盖不足的部分
4. **结果综合**：将图谱查询的精确结果和向量检索的语义结果合并

混合查询的优势是同时具备了图谱查询的精确性和向量检索的覆盖面。

## 13.7 本章小结

知识图谱构建是一个系统工程，从 Schema 设计到实体抽取、关系抽取、质量检查再到混合查询，每个环节都需要精细的设计和持续的优化。本章介绍了基于词典的实体抽取、基于触发词的关系抽取、Schema 约束和质量检查机制，以及最终的 Graph+Vector 混合查询方案。

完整的可运行代码示例见 `demos/ch13/kg_construction.py`，它演示了从词典实体抽取、规则关系抽取、图谱存储到质量检查和混合查询的完整流程。
